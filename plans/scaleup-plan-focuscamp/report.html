focus.camp scale-up plan

Kế hoạch nâng focus.camp từ single-VPS early production lên multi-instance production.

Bản HTML này tóm tắt kế hoạch update code và hạ tầng để hệ thống scale được: stateless app replicas, Redis bắt buộc, worker singleton, cache/counter/index cho hot path, realtime SSE, service-side permission và deploy an toàn hơn.

Đọc nhanh: làm cái nào trước?

Câu trả lời ngắn là làm security trước, chưa scale hạ tầng ngay.

1

PR đầu tiên: `security-service-hardening`

Fix các cửa bypass quyền ở backend/service trước: chat, report, API key, owner row, best-answer, product purchase, challenge payment.

2

Vì sao không làm infra trước?

Nếu thêm nhiều app replicas trước khi khóa permission, lỗi bảo mật sẽ bị nhân rộng, khó trace hơn và dễ thành incident hơn.

Làm xong thì thấy gì?

User bị remove/demote không còn dùng API key, chat, report, purchase, challenge path để bypass được.

Sau đó làm gì?

Redis required + distributed rate limit, rồi DB index/counter/cache, rồi realtime SSE, worker, deploy multi-instance.

Câu nhớ nhanh

Khóa cửa trước, rồi mới mở thêm nhà. Engineering translation: secure service boundaries first, then scale infrastructure.

Thứ tự Việc Mục tiêu dễ hiểu
1 Security fixes Không ai bypass quyền bằng API/action trực tiếp.
2 Redis + rate limit Nhiều server vẫn dùng chung giới hạn request và trạng thái realtime.
3 DB index + counter + cache Trang đông người không query nặng lặp lại liên tục.
4 Realtime SSE Chat/notification cập nhật ngay, không phải refresh/poll DB quá nhiều.
5 Worker + cron split Job nền không bị chạy trùng khi có nhiều app replicas.
6 Multi-instance deploy Lúc này mới mở rộng app replicas sau load balancer.

Trạng thái hiện tại

Codebase đã đủ nền sản phẩm, nhưng các đường vận hành scale cần được khóa lại trước khi tăng traffic.

Đã có nền tốt

Next.js App Router, Prisma/Postgres, Redis/BullMQ optional, R2/S3, SePay, NextAuth, AI Agent/MCP.

Chưa replica-safe

Rate limit fallback in-memory, worker/cron có nguy cơ chạy lặp, DB/Redis còn gắn với một stack.

Cần chốt service layer

Một số flow vẫn phụ thuộc UI hoặc caller-provided id. Khi scale, các gap permission sẽ thành incident.

Kết luận ngắn Scale được, nhưng cần đi theo thứ tự: security -> Redis/rate limit -> DB/cache -> realtime -> worker/infra.
Không làm Không rewrite app, không chuyển khỏi Postgres, không nhảy thẳng Kubernetes, không sửa app code trong artifact này.

Kiến trúc mục tiêu

App replicas stateless, mọi state phân tán đi qua Postgres/Redis/R2, worker chạy riêng.

6 track scale chính

Thứ tự ưu tiên giữ cho hệ thống không phình rủi ro trong lúc tăng tải.

T1

Multi-instance Infrastructure

Tách app khỏi DB/Redis, thêm health live/ready, deploy image theo SHA, worker singleton.

  • Tách `docker-compose.yml` local/dev và `docker-compose.prod.yml` production.
  • App replicas dùng `WORKER_ENABLED=false`; worker service riêng dùng `WORKER_ENABLED=true`.
  • `/api/health/live` chỉ check process; `/api/health/ready` check DB, Redis, env critical.
T2

Database, Query, Counter, Cache

Giảm tải hot path bằng index đúng query shape, counter denormalized và Redis cache có invalidation.

  • Ưu tiên shell, discovery, community summary, feed, challenge review counts, marketplace lists.
  • Thêm counters như `Post.commentCount`, `Post.reactionCount`, `Challenge.participantCount` nếu schema chưa có.
  • Không cache mù: mỗi cache key phải có điểm invalidate khi mutation xảy ra.
T3

Realtime Chat And Notifications

Dùng SSE v1 qua Redis pub/sub cho notification unread count và chat message, WebSocket để sau.

  • Thêm `GET /api/chat/channels/[channelId]/stream` và API message cursor.
  • Service chat phải check ACTIVE membership và channel ownership trước khi send/stream.
  • Client reconnect thì refetch cursor để không mất message.
T4

Redis, Rate Limit, Queue, Worker

Redis production là bắt buộc, rate limit distributed, queue idempotent, worker không chạy lặp.

  • `REQUIRE_REDIS=true` làm production fail readiness nếu Redis thiếu.
  • Rate limit dùng Redis sliding window/token bucket, chỉ local/dev mới fallback memory.
  • Cron cleanup, AI review, notification fanout chạy qua worker/queue.
T5

Security And Permission Hardening

Fix các permission gap trước khi scale vì scale sẽ khuếch đại lỗi bảo mật.

  • Không tin `communityId`, `channelId`, `postId`, `challengeId` từ client nếu derive được từ DB.
  • API key request phải revalidate current member role/status sau demotion/removal.
  • Owner row, report resolve, best-answer, product purchase, challenge payment đều cần service-side guard.
T6

Service Layer Cleanup, Testing, Observability

Business logic đi qua `lib/services/`; action/API chỉ parse session, validate input, gọi service.

  • Audit `rg "prisma\." app lib components` để chuyển logic rủi ro về service.
  • Thêm test service integration cho role, payment, challenge, report, chat.
  • Log metrics: route duration, Redis latency, queue lag, SSE connections, rate-limit denials.

Release

Rủi ro P0 cần xử lý trước scale

Các lỗi này cần fix ở service/backend, không chỉ hide UI.

Rủi ro Mức Hướng fix Verify
Chat send/stream thiếu membership và channel ownership recheck. High Service derive community từ channel, check ACTIVE membership trước DB write/stream. Removed member không send/stream được.
Report resolution tin `communityId` caller-provided. High Load report by `reportId`, derive community từ DB, verify actor permission. Wrong communityId không bypass được.
API keys có thể sống sau demotion/removal. High Mỗi request revalidate current membership role/status và key scope. Demote/remove xong request tiếp theo bị reject.
`updateMemberRole` thiếu owner-row protection ở service. High Không cho update/remove owner row, owner transfer là flow riêng. ADMIN/MOD/action direct không demote OWNER được.
Product purchase membership enforcement chưa rõ ở service. Medium `startProductPurchase` verify product/community access, không dựa UI disabled. Non-member gọi API trực tiếp bị deny.
Challenge AIP payment cần verify challenge-community relation. Medium Payment target phải derive và assert cùng community/challenge. Mix challengeId/communityId sai bị reject.

Definition of Done

Điều kiện đóng phase và đóng toàn bộ scale plan.

Mỗi phase xong khi

  • Code change nhỏ, phạm vi rõ.
  • Có migration/backfill nếu đổi schema.
  • Có test cho behavior rủi ro.
  • Có manual verify command/result.
  • Có rollback note nếu deploy production.

Toàn plan xong khi

  • App chạy 2+ replicas với shared DB/Redis.
  • Redis bắt buộc trong production.
  • Worker/cron không duplicate.
  • Hot routes có cache/index/counter.
  • 7 security risks đã fix và có test.
Nguyên tắc triển khai: permission fix phải service-side, cache phải có invalidation, queue job phải idempotent, và khi sửa Next.js behavior phải đọc local `node_modules/next/dist/docs/` vì repo dùng Next 16.
